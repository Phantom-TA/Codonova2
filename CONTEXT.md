# Codonova Autonomous System Context Board

This file provides comprehensive context regarding the **Codonova** autonomous AI software development system. It acts as the single source of truth for any LLMs or developers maintaining the system.

## 1. System Architecture
Codonova is designed to automatically ingest user requirements, decompose them into a project plan using Neo4j as a graph representation, and autonomously generate, test, and correct code via a loop of specialized AI Agents.

### Tech Stack
- **Backend:** Python + FastAPI
- **Frontend:** React + Nginx Proxy
- **Databases:** Neo4j Database (Task Graph & Agent Metrics), ChromaDB (Vector Search / Memory)
- **Deployment:** Docker Compose (services: `backend`, `frontend`, `neo4j`, `chromadb`)

---

## 2. The Autonomous AI Agents
The engine lives inside `/backend/agents/`. The backend acts as the orchestrator for multiple prompt-driven agents:
1. **PlanningAgent:** Receives human input, designs the architecture, extracts features, and decomposes it into dozens of micro-tasks linked via dependency edges (`DEPENDS_ON`) in Neo4j.
2. **DeveloperAgent:** Generates code for specific tasks using the Graph database task dependencies as context.
3. **TestingAgent:** Writes unit tests for the output generated by the Developer Agent.
4. **EvaluatorAgent:** Grades the tests.
5. **DebuggingAgent:** Re-analyzes failed test outputs and corrects the original code.

**Self-Correction Engine:** If an agent creates code that fails its tests, it triggers a 3-retry correction loop across the Test/Debug/Dev agents before blocking the task sequence entirely.

---

## 3. LLM API & Routing
Routing code lives in `/backend/llm_client.py`.
The system dynamically routes requests based on task complexity:
- **`reasoning` agent:** Uses Google Gemini (Default: `gemini-2.5-flash`). Used for complex tasks (planning, developing, debugging).
- **`fast` agent:** Also dynamically routes to Gemini (historically Groq, but Groq's 12K TPM limit breaks on context-heavy tests). Used for high-volume rapid evaluations.
- **Fallback (`OpenRouter`):** Protects the system against temporary API provider outages. Uses `qwen-2.5-7b-instruct:free` on OpenRouter.

### **Important Resilience Features:**
- **Rate Limit Survival:** Because Gemini Free Tier is restricted to 15 Requests Per Minute (RPM), `llm_client.py` has a hardened retry block. Upon hitting an HTTP 429 quota exception, the system uses up to **5 retries** structured with **30-second exponential delays**, easily wrapping around the minute timeout threshold without crashing.
- **UI Proxy Survival:** Because agent completions can take multiple minutes synchronously due to rate limit sleeping, the `/frontend/nginx.conf` proxy is configured with a custom `proxy_read_timeout 600;` (10 minutes) for `/api/` traffic so HTTP requests do not 504 prematurely.

---

## 4. Neo4j Graph Hierarchy
Codonova is inherently graph-driven. System state is managed via Cypher queries (`neo4j_client.py`) rather than REST or relational tables.

- **Nodes:** `Project`, `Feature`, `Task`, `CodeModule`, `Agent`
- **Edges:** `HAS_FEATURE`, `HAS_TASK`, `DEPENDS_ON`, `PRODUCED_BY`

Tasks go through a specific lifecycle (`status`): `PENDING` → `IN_PROGRESS` → `DONE` / `FAILED` / `BLOCKED`.
Agent profiles are dynamically tracked (Rolling Average tracking over Total Tasks execution).
